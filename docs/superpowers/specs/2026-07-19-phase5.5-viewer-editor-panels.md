# Part 1 вҖ” Design Spec

## 1. Summary

Phase 5.5 turns MatterViewer from a debug HUD with a world picker into an actual scene editor. Every data model this phase needs already exists and is unit-tested вҖ” `EditorModel`, `PropertiesRegistry`, `SpecializedEditors`, `SimulationControl`, `DynamicSceneBridge`, `part_graph_snapshot::Snapshot` вҖ” but none of it is wired to ImGui. This phase builds the UI layer on top of those models:

- A **unified scene tree** panel listing baked geometry roots, ECS entities, and runtime-spawned objects in one hierarchy.
- A **properties inspector** with auto-generated widgets per component, specialized inline editors, and read-only baked-root info.
- **Viewport picking** (ECS entities and baked instances), **multi-select**, and a **selection outline**.
- A **transform gizmo** synced with the `LocalTransform` fields in the inspector.
- A **simulation toolbar** (Play/Pause/Step/Stop) driving `SimulationControl`, with visual mode feedback.
- A **console/log panel** for bake errors and diagnostics.
- **ImGui docking** with one shipped default layout.

The deliverable is an editor loop: select anything in the scene, inspect and edit it, press Play to simulate, press Stop to restore вҖ” all without leaving the viewer.

## 2. Motivation

Phase 5 wired authored JS entities into ECS instantiation, and Phase 4 built the render bridge and snapshot/restore machinery. The engine can now *represent* an editable scene, but the only way to inspect it is print statements and the debug HUD. Iteration on `PhysicsPlayground`-style worlds currently means: edit JS, relaunch, squint at the viewport. An entity list, an inspector, and play-mode controls close that loop and are prerequisites for Phase 6 (gameplay scripting), which needs a console and a live view of runtime-spawned entities to be debuggable at all.

The models were deliberately built UI-free (callback-driven `SceneCommands` / `ComponentCommands` structs) so this phase is almost entirely additive: new `Ui` draw methods plus a handful of small engine-side exposure gaps (`graph_snapshot` through `WorldSession`, pick readback, outline rendering).

### 4.1 Scene Tree panel

- Header: text filter box (substring match on name/id, backed by `EditorModel::set_filter`) plus a filter-mode combo: **All / Entities only / Roots only**.
- Node rendering per row kind:
  - `[Root]` вҖ” baked geometry root from `graph_snapshot()` nodes with `is_root == true`. Icon color: blue. Children under it (optional expansion) come from `Node::children` / `walk_part_tree`.
  - `[Entity]` вҖ” ECS entity from `EditorModel::rows()` (`HierarchyRow`), indented by `depth`, expandable if `child_count > 0`.
  - `[Part]` вҖ” leaf under an entity that has a `PartInstance`: shows the resolved part name. Selecting it selects the *owning entity* (parts are not independently selectable).
  - `[Runtime]` вҖ” entity whose `SceneEntityId` is absent from the last Edit-mode snapshot (i.e., spawned during Play). Same behavior as `[Entity]` but visually distinct (italic / dim).
- Selection highlight follows the shared selection set (В§5). Clicking behaviors per D3.
- Right-click opens the context menu (В§4.5) on entity rows only.

### 4.2 Properties panel

**ECS entity selected (single):**
- Header: entity name (editable text field вҶ’ rename), id in hex, parent name.
- One `CollapsingHeader` per component present on the entity, in `PropertiesRegistry::entries()` order. Fields rendered from `FieldWidget` metadata:
  - `FloatSlider`/`IntSlider` вҶ’ `SliderFloat`/`SliderInt` with `range_min/max`
  - `FloatDrag`/`IntDrag`/`UIntDrag` вҶ’ `DragFloat`/`DragInt`/`DragScalar`
  - `Checkbox` вҶ’ `Checkbox`
  - `EnumDropdown` вҶ’ `Combo`
  - `Float3Editor` вҶ’ three-wide `DragFloat3`
  - `QuaternionEditor` вҶ’ euler-angle proxy (drag XYZ degrees, converted to/from quaternion; raw quat shown in a tooltip)
- Components with `has_specialized_editor(kind) == true` (PartInstance, RigidBody, SectorStreaming) additionally render their specialized controls inline *inside* the same section (В§4.3).
- Removable components (`user_removable`) get an `вң•` button on the header.
- Footer: `[+ Add Component]` popup listing `addable_components(record)`.

**ECS entities selected (multi):**
- Header: "N entities selected".
- Sections only for components present on *every* selected entity. A field whose value differs across the set renders with an empty/placeholder value ("вҖ”"); committing a value writes it to all selected entities. Specialized-editor action buttons apply to all.
- Add Component adds to all; Remove removes from all that have it.

**Baked root selected:**
- Read-only info card from the `part_graph_snapshot::Node`:
  - Module name, `source_path`, `resolved_hash` (hex, click-to-copy)
  - `params_json` (collapsed, pretty-printed)
  - Stats: direct child count (`children.size()`), poly/instance count aggregated via `walk_part_tree` over the part store
  - `[Open Source]` button вҶ’ shell-open `source_path` in the OS default editor (`ShellExecuteA` on Windows, `xdg-open` on Linux)

### 4.3 Specialized editor sections (inline)

- **PartInstance**: resolved part name (hash вҶ’ name via `PartEditorCommands::list_available_parts`), `[Pick PartвҖҰ]` popup driving `assign_part`, visible / cast-shadow checkboxes (these map to the registry fields but the section owns their layout).
- **RigidBody**: mass and velocity fields from the registry, plus action buttons: `[Wake]`, `[Set Velocity]`, `[Apply Impulse]` (small inline Float3 + button), `[Teleport To Camera]` вҖ” wired to `PhysicsEditorCommands`.
- **SectorStreaming**: attach/detach button, radius field, follow-camera checkbox, seed + `[Regenerate]` вҖ” wired to `StreamingEditorCommands` / `StreamingEditorState`. This completes the Phase-4 retirement of the standalone sector panel.

### 4.4 Toolbar

- Left-aligned transport buttons. Enabled/disabled state derives strictly from `SimulationControl::mode()`:
  - Edit: Play enabled; Pause/Step/Stop disabled
  - Play: Pause, Stop enabled; Play/Step disabled
  - Pause: Play (resume), Step, Stop enabled
- Center: mode label ("EDIT" / "PLAYING" / "PAUSED").
- Viewport feedback: a 3px inner border drawn over the viewport region via the ImGui foreground draw list вҖ” green in Play, amber in Pause, none in Edit.

### 4.5 Tree context menu (entity rows only)

| Item | Action | Backing call |
|------|--------|--------------|
| Focus (F) | Frame camera on node bounds | shared focus routine (В§5.4) |
| Add Child Entity | Spawn empty entity parented to this node | `SceneCommands::create_empty` + `reparent` |
| Duplicate | Clone with `+0.5m` X offset on the copy's LocalTransform | `SceneCommands::duplicate`, then transform nudge |
| Delete | Destroy entity (and ECS children) | `SceneCommands::delete_entity` |

Baked roots get only **Focus** and **Open Source**. Destructive items are disabled during Play (allowed in Pause and Edit).

### 4.6 Console panel

- Ring buffer (default 4096 entries) of `LogEntry { severity, time, message }`.
- Sources wired in this phase: world connect/bake results (parts baked, cache hits), bake errors from the provider, `SceneEditResult` failures, bridge `PartInstanceError` reports, simulation transition errors (the `std::string& error` out-params).
- Severity filter checkboxes, text filter, Clear button, auto-scroll (disengages when the user scrolls up, re-engages at bottom).
- Phase 6 will add a script input line at the bottom; the panel is designed with a reserved footer row.

## 5. Interaction Model

### 5.1 Selection data model

A new `SelectionSet` (viewer-side) replaces `EditorModel`'s single-slot selection as the source of truth:

```cpp
struct SelectedObject {
    enum class Kind : uint8_t { Entity, BakedRoot } kind;
    matter::scene::SceneEntityId entity_id{};  // Kind::Entity
    std::string root_module;                   // Kind::BakedRoot
};
class SelectionSet {
    void replace(SelectedObject);            // plain click
    void toggle(SelectedObject);             // Ctrl+click
    void extend_range(...);                  // Shift+click (tree order)
    void clear();
    // validation against world generation; invalid ids dropped on refresh
};
```

`EditorModel::select/clear_selection` remain for compatibility and mirror the *primary* (last-clicked) entity so existing model code keeps working. Selection is validated each frame against `SceneCommands::generation()`; entities that vanished (deleted, or destroyed by Stop-restore) are silently dropped from the set.

### 5.2 Sources of selection

1. **Tree click** вҖ” with Shift/Ctrl modifiers per D3. Range select uses visual (filtered, flattened) row order.
2. **Viewport click** вҖ” pick resolution per В§6. Shift/Ctrl work identically. Click on empty space with no modifier clears.
3. **Programmatic** вҖ” Stop clears (D12); world switch clears.

### 5.3 Hotkeys (active when viewport or tree is focused, never during text input)

| Key | Action |
|-----|--------|
| F | Frame selection |
| R | Gizmo вҶ’ rotate |
| S | Gizmo вҶ’ scale |
| G / T | Gizmo вҶ’ translate |
| Delete | Delete selected entities (Edit/Pause only) |
| Esc | Clear selection |
| Space | Play/Pause toggle |

### 5.4 Focus / frame

Compute a merged world-space AABB of the selection (entity: part AABB from the part store transformed by WorldTransform, or a 1m default cube for part-less entities; baked root: sector-grid bounds for that root's instances). Animate the orbit camera target to the AABB center and dolly so the AABB fits at ~35В° vertical FOV coverage, over ~0.25s.

### 5.5 Gizmo

- Rendered as a viewport overlay (ImGuizmo vendored into `Libraries/imgui/`, or hand-rolled axis handles if vendoring is rejected during implementation вҖ” ImGuizmo is the plan of record).
- Operates on the primary selection's **LocalTransform**; for multi-select, translates all selected by the same world-space delta (rotate/scale apply per-object about their own pivots вҖ” matching Blender "individual origins" as the simple default).
- Manipulation writes LocalTransform through the same mutation path the properties panel uses, so both stay in sync trivially (both read the component fresh each frame in Edit/Pause).
- Hidden and inert while `mode() == Play`.

## 6. Picking & Highlight

### 6.1 ECS entity picking (dynamic instances)

The Vulkan pipeline already writes a per-instance history token; `DynamicSceneBridge::resolve_pick(uint32_t token)` maps it back to a `SceneEntityId`. Missing piece: a clickвҶ’token readback. Implementation: a 1Г—1 pick request вҖ” on click, record the cursor position; the next frame's cull/raster pass writes the visible instance token at that pixel into a small readback buffer (or, first iteration, a CPU ray vs. tracked-entity OBB test using `DynamicSceneBridge::scene_entities()` + part AABBs, which needs zero GPU work). **Plan of record: start with the CPU ray test (correct and simple at editor entity counts), keep the GPU token path as the noted upgrade.** `resolve_pick` is still used to classify GPU-path results when that lands.

### 6.2 Baked geometry picking

CPU raycast: camera ray вҶ’ sector grid cells along the ray вҶ’ per-instance transformed AABB test вҶ’ nearest hit's part hash вҶ’ owning root module via reverse lookup built from `graph_snapshot()` (`walk_part_tree` from each root records which hashes belong to which root; cache this map per world connect). Precision beyond AABB (triangle-accurate) is out of scope.

### 6.3 Selection outline вҖ” research directions (Task 6 resolves)

Investigated options against the existing renderer (`renderer.h` / `raster_composer.h`, GL raster path + Vulkan path):

1. **Stencil outline** (preferred, raster path): re-draw selected instances with stencil write, then a second pass scaled ~1.02 along normals with stencil test inverted, flat highlight color. Requires a small "selected instance list" uniform path into the composer. Cheap, crisp, standard.
2. **Post-process edge detect** on an id/depth buffer: best quality, but needs an id attachment through both GL and Vulkan paths вҖ” too invasive for this phase.
3. **Wireframe AABB overlay** (fallback, ships first): draw selected objects' world AABBs as line lists via the existing debug-draw hooks. Zero renderer surgery, works identically in both backends, always correct.

**Decision: ship (3) as the Task-6 baseline; implement (1) in the same task if the raster-path stencil plumbing proves < ~1 day, otherwise file it as follow-up.** The spec's acceptance criteria are written against (3); (1) is a quality upgrade, not a gate.

The properties panel shows a small "changes made in Pause are lost on Stop" hint the first time a Pause edit occurs.

## 8. Unified tree rationale

Three object populations coexist at runtime: baked static geometry (sector-instanced, no ECS identity), authored ECS entities, and play-spawned runtime entities. Splitting these across panels forces users to know engine internals to find an object. The unified tree presents provenance as a *tag*, not a *location*:

- One search box finds everything.
- Selection semantics are uniform; the properties panel branches on kind.
- Baked roots being read-only in the tree makes the authoring contract visible: *static world content is edited in JS and rebaked; dynamic content is edited live* вҖ” the `[Open Source]` button is the bridge between the two workflows.
- `[Runtime]` tagging gives immediate visibility into what Play created, and previews exactly what Stop will destroy.

Data sources are merged at draw time, not stored in a merged model: baked roots from `WorldSession::graph_snapshot()` (cached per connect, refreshed on live-edit reresolve), entities from `EditorModel` (refreshed per frame from `SceneCommands`). This keeps `EditorModel` untouched and unit-testable as-is.

## 9. Non-goals (deferred)

Layout presets and switching В· viewport right-click menus В· asset browser and drag-drop part assignment В· undo/redo В· copy/paste entities В· game-camera (non-editor) viewport В· triangle-accurate picking В· timestep configuration UI В· Linux `ui_linux` parity beyond compiling (panels are shared where the code allows; polish is Windows-first this phase).

# Part 2 вҖ” Implementation Spec

## Conventions

- New viewer files live in `MatterViewer/`; engine-side changes stay behind `WorldSession` / existing headers.
- Every task ends with: `make -C MatterViewer windows` links clean, and where a headless test is listed, it runs via `make -C MatterEngine3/tests run-<name>` (TEMP/TMP vars per CLAUDE.md).
- UI tasks that can't be headless-tested get a **manual acceptance** checklist instead (pattern: `2026-07-18-sector-streaming-manual-acceptance.md`).
- All tasks keep `ui_linux.cpp` compiling (stub or share code; parity polish deferred per В§9).

## Task list

### Task 1 вҖ” ImGui docking + default layout scaffolding

**Build:** Enable `ImGuiConfigFlags_DockingEnable` in `Ui::setup`.
